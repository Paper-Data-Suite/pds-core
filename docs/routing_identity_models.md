# PDS2 Routing Identity Models

PDS Core implements the shared identity and registration values established by
[ADR 0001](decisions/0001-adopt-pds2-page-locator-routing.md) in
`pds_core.routing_models`. Route IDs are generated by `pds_core.route_ids`.

The contract separates five concepts:

```text
ModuleWorkRef
    -> RouteLocator
        -> RouteRegistration
            -> ModuleRecordRef target

RouteResolution = validated locator + registration + calculated roots
```

A valid value establishes structure and identity only. It does not establish
that a class, work, registration target, or path exists, and it does not
authorize evidence processing or scoring.

## Identity composition

`ModuleWorkRef` identifies one module-owned work context:

```text
module_id + class_id + work_id
```

`RouteLocator` adds the non-semantic identifier of one expected returned
physical page route:

```text
module_id + class_id + work_id + route_id
```

`ModuleRecordRef` separately identifies the module-owned record to which a
route is registered. Route and target identity are deliberately distinct. A
registration requires its locator and target to have the same `module_id`.

All identifiers use Core's safe identifier character set: ASCII letters,
digits, `_`, and `-`. Values must be non-empty strings with no leading or
trailing whitespace. `module_id` and `record_kind` must also be lowercase;
Core rejects rather than normalizes mixed-case values.

## `ModuleWorkRef`

`ModuleWorkRef` is a frozen, slotted, hashable value object. It is not limited
to assignments, assessments, student work, or gradebook-reportable work.

Its exact mapping shape is:

```json
{
  "module_id": "concord",
  "class_id": "english10_p3",
  "work_id": "socratic_seminar_1"
}
```

## `RouteLocator`

`RouteLocator` is the normalized in-memory identity represented by a PDS2 QR
payload. It requires `schema == "PDS2"`, a validated `ModuleWorkRef`, and a safe
`route_id`. Its read-only `module_id`, `class_id`, and `work_id` properties
delegate to the composed work reference.

The mapping representation is intentionally flat:

```json
{
  "schema": "PDS2",
  "module_id": "concord",
  "class_id": "english10_p3",
  "work_id": "socratic_seminar_1",
  "route_id": "rt_0123456789abcdef0123456789abcdef"
}
```

The locator does not contain a student, Author, Subject, logical page number,
destination path, Score target, or arbitrary QR metadata.

## `ModuleRecordRef`

`ModuleRecordRef` is a generic typed reference to a module-owned record. Core
validates its structure but does not interpret `record_kind` or look up the
record. Representative kinds include `answer_sheet_page`, `response_page`, and
`artifact_page`.

Its exact mapping shape always contains `contract_version`, including when the
value is null:

```json
{
  "module_id": "concord",
  "record_kind": "artifact_page",
  "record_id": "artifact_page_0123456789abcdef",
  "contract_version": "1"
}
```

`contract_version` identifies a module-record contract, not an installed
package version. When present, it must be a safe identifier such as `1` or
`v1`; dotted package versions are not accepted by the current identifier
contract.

## `RouteRegistration`

`RouteRegistration` is the Core-readable persisted relationship between one
locator and one target. It performs no file access. Its exact seven-key shape
is:

```json
{
  "schema_version": "1",
  "locator": {
    "schema": "PDS2",
    "module_id": "concord",
    "class_id": "english10_p3",
    "work_id": "socratic_seminar_1",
    "route_id": "rt_0123456789abcdef0123456789abcdef"
  },
  "target": {
    "module_id": "concord",
    "record_kind": "artifact_page",
    "record_id": "artifact_page_0123456789abcdef",
    "contract_version": "1"
  },
  "created_at": "2026-07-14T09:00:00-04:00",
  "status": "active",
  "human_fallback": "PDS2 | concord | english10_p3 | socratic_seminar_1 | rt_0123456789abcdef0123456789abcdef",
  "module_details": {}
}
```

The registration requires:

- schema version `1`;
- validated locator and target models;
- matching locator and target module ownership;
- a valid timezone-aware ISO 8601 creation timestamp;
- a shared registration status;
- a non-empty, trimmed, single-line human fallback without control characters;
- and finite JSON-compatible `module_details` with string mapping keys.

`module_details` may carry lightweight lookup information or diagnostics, but
the module-owned record remains authoritative. Core recursively freezes a
private copy of the supplied details. The public `module_details` property and
serialization each return fresh JSON-native dictionaries and lists, so direct
JSON serialization succeeds and later caller or output mutation cannot change
the registration.

### Registration statuses

The initial shared statuses are:

| Status | Meaning |
| --- | --- |
| `active` | Eligible for normal module handling. |
| `inactive` | Intentionally unavailable without a more specific terminal classification. |
| `retired` | Historical and no longer expected in the active workflow. |
| `superseded` | Replaced by a later route or page. |
| `cancelled` | Deliberately cancelled. |
| `invalidated` | Erroneous, unsafe, or otherwise invalid for processing. |

Non-active registrations remain structurally resolvable so later workflow code
can explain their state. These models do not implement status transitions or
authorization policy.

## `RouteResolution`

`RouteResolution` is a runtime-only frozen, slotted value. It pairs a locator
with the exact matching registration and three Core-calculated `Path` values:
`class_root`, `module_root`, and `work_root`.

Construction validates types and exact locator equality only. It does not
create paths, resolve symlinks, require paths or targets to exist, load module
records, or perform any filesystem I/O. A non-active registration can be
represented by a resolution. The owning module still decides whether its
target and workflow permit further handling.

There is intentionally no shared `RouteResolution` JSON schema or conversion
function.

## Exact mapping conversion

The four persisted identity models provide `validate_*`, `*_to_dict`, and
`*_from_dict` functions. Validators accept either an already constructed model
or a mapping and return the validated model. Mapping input must have exactly
the documented keys. Missing, unknown, non-string, and invalid nested keys are
rejected with `RoutingModelError`; aliases and legacy fields are not normalized.

Serialization returns new ordinary dictionaries containing only JSON values.
It never returns live mutable registration details.

```python
from pds_core.routing_models import (
    route_registration_from_dict,
    route_registration_to_dict,
)

registration = route_registration_from_dict(registration_data)
safe_json_data = route_registration_to_dict(registration)
```

## Route-ID generation

`generate_route_id()` takes no semantic input and uses standard-library secure
randomness to return:

```text
rt_<32 lowercase hexadecimal characters>
```

The current format contains 128 random bits and is practically globally
collision-resistant. Generation performs no filesystem lookup; exclusive
registration creation and persisted collision handling are separate concerns.

`RouteLocator` accepts any route ID satisfying the shared safe identifier
contract. It does not require IDs to match the current generator format, which
allows future compliant generators without changing PDS2 identity semantics.

## Ownership boundaries and legacy formats

Core owns the generic identity structures, exact registration schema,
structural validation, and route-ID generator. Modules own the meaning and
existence of work and target records, semantic relationships, evidence
destinations, and scoring or processing authorization.

These models do not inherit from or convert through the legacy `QrPayload`.
PDS1 and OMR1 have incompatible student-, assignment-, and page-oriented
semantics and are not accepted by this API. Their temporary implementation
elsewhere in Core is not part of the PDS2 routing identity contract.
